# 虚幻造物 v7.0 — 最终推荐融合方案总规划（newplan4）

> 版本：7.0
> 日期：2026-03-06
> 文档定位：当前最高优先级规划主文档
> 融合来源：`newplan3.md` + `claude-plan.md`
> 核心目标：在不牺牲系统可控性的前提下，让用户把注意力集中在内容创作、质量判断与导演级决策上，逐步逼近“奥斯卡电影级 / 国内顶级短剧级”生产标准。

---

## 一、文档定位与最终决策

本版 `newplan4.md` 是在 `newplan3.md` 与 `claude-plan.md` 基础上的最终推荐融合方案。

两者在项目中的角色重新定义如下：

- `newplan3.md`：提供系统主骨架、工程完整度、资产库、导出链、数据模型、API、风险治理与长期可控性
- `claude-plan.md`：提供最优的前台交互范式、三段式认知模型、心流设计与中段去节点化的产品形态判断
- `newplan4.md`：把前者的“系统可控性”与后者的“创作心流”合并成一个面向最终产品的统一主规划

### 1.1 最终推荐结论

最终不采用以下两种单独路线：

- 不单独采用 `newplan3.md` 作为前台产品形态，因为它的中段仍偏工程系统心智，可能影响创作心流
- 不单独采用 `claude-plan.md` 作为完整落地蓝图，因为它的前台体验很强，但后台规格、治理、交付、风控与系统骨架不够厚

最终采用的路线是：

**前台产品形态采用 `claude-plan.md` 的三段式创作引擎；后台系统骨架采用 `newplan3.md` 的完整工作流、资产、导出、数据和治理体系。**

### 1.2 一句话定位

**工坊写故事，画板做镜头，预演看成片；Pipeline 藏在引擎盖下，质量系统贯穿全过程。**

### 1.3 最终平台形态

最终平台不是“一个大画布”，而是：

- Stage 1：`Workbench` 工坊 —— 负责语言思维与叙事创作
- Stage 2：`Board` 画板 —— 负责空间思维与镜头/素材编排
- Stage 3：`Preview` 预演 —— 负责时间思维与审片/导出
- Hidden Engine：`Pipeline + Agent + Asset + Scoring + Governance` —— 作为底层执行与可控性系统存在

## 二、核心融合原则

### 2.1 前台创作优先，后台系统兜底

用户看到的主界面必须服务于创作，不服务于系统结构展示。
系统复杂性要尽量沉到后台和抽屉视图中，由模板、默认值、质量守护、预算规则和推荐流程吸收。

### 2.2 三段式认知模型优先于单一画布模型

影视创作包含三类认知活动：

1. 语言思维：故事、角色、对白、情绪、叙事节奏
2. 空间思维：镜头、场景、构图、风格、视觉联动
3. 时间思维：剪辑、音乐、对白节奏、转场、观看体验

所以产品必须让用户在三种最适合的工作台中切换，而不是在一个万能界面中受困。

### 2.3 节点系统继续存在，但默认不暴露

节点、连线、Agent 任务、执行链、回写、审计都继续保留，因为它们是平台可控性的基础。
但它们不应继续作为默认主界面。

最终原则是：

- 默认主界面：卡片、内容块、候选结果、时间线、评论、评分
- 高阶可见层：Pipeline 抽屉 / Layer 2 流程视图
- 系统层：工作流引擎、AgentBus、Artifact、WriteBack、审计、预算治理

### 2.4 心流优先，不等于弱化专业度

真正的专业度不是让用户操作更多控件，而是：

- 用户能更长时间持续聚焦在人物、冲突、镜头、节奏、声音和风格上
- 系统能在背后提供更强的可控性、更强的一致性、更强的回写与复盘能力

### 2.5 质量比生成更重要

最终目标不是“生成很多内容”，而是：

- 把差的结果淘汰掉
- 把好的结果保留下来并升级
- 把关键判断过程可视化和结构化
- 把质量控制做成系统，不靠运气

## 三、最终产品架构

### 3.1 前台三段式创作引擎

#### Stage 1 — 工坊 Workbench

- 核心界面范式：`Sudowrite` 式文档流创作工作台
- 认知模式：语言思维
- 基本操作单元：`Beat` / `Scene`
- 用户目标：把故事、人物、冲突、对白、情绪写清楚

#### Stage 2 — 画板 Board

- 核心界面范式：`Tapnow/TapFlow` 式可执行创作画板
- 认知模式：空间思维
- 基本操作单元：`Shot Card`
- 用户目标：把 Scene 转成 Shot 方案，把视觉资产和参考放进镜头创作空间中，把候选方案进行比较与筛选

#### Stage 3 — 预演 Preview

- 核心界面范式：`Premiere/CapCut` 式预演与交付工作台
- 认知模式：时间思维
- 基本操作单元：`Shot / Clip / Timeline Segment`
- 用户目标：看片、批注、回跳修正、再看片，并最终导出交付包

### 3.2 隐藏系统层

作为三段式背后的核心操作系统，平台保留以下系统层：

- Workflow / Pipeline 引擎
- Agent 运行时与事件总线
- Asset Library
- Quality Scoring
- Candidate Arena
- Director Review Loop
- Performance & Emotion Control Layer
- Budget & Rerun Governance
- Artifact / Version / WriteBack / Audit
- Export / CapCut Draft / Marketing Package

## 四、四层用户群体版本体系

### 4.1 `Normal` — 入口层

- 用户：个人创作者、编剧新手、轻量试用者
- 核心诉求：低门槛从小说/剧本到分镜草案与简化预演
- 主要体验：工坊主导 + 轻量画板 + 简化预演
- 原则：不暴露 Pipeline 复杂度，不要求理解工作流系统

### 4.2 `Canvas` — 生产层

- 用户：短剧工作室、影视公司内容团队、专业编剧协作组
- 核心诉求：更快生产、更高质量、更强协作、更少工具负担
- 主要体验：完整三段式，其中画板是主力生产台
- 原则：默认以卡片画板操作，Pipeline 可见但不要求直接操控

### 4.3 `Hidden` — 方案层

- 用户：合作伙伴、联合项目组、标杆客户
- 核心诉求：高阶控制、定制知识库、Cinema Lab、复杂方案与质量策略
- 主要体验：完整三段式 + 高阶控制视图 + 更深的系统可见性

### 4.4 `Ultimate` — 内部操作系统层

- 用户：内部研发、导演实验室、算法调优、运营分析
- 核心诉求：实验、调试、评测、AB 策略、系统治理、数据闭环
- 主要体验：全量三段式 + 全量 Pipeline + 调试与观测系统

### 4.5 四版本 × 三段式能力矩阵

| 能力 | Normal | Canvas | Hidden | Ultimate |
|---|---|---|---|---|
| 工坊写作 | 完整 | 完整 | 完整 | 完整 |
| 画板默认视图 | 简化卡片板 | 完整卡片板 | 完整卡片板 | 完整卡片板 |
| Pipeline 抽屉 | 不开放 | 可见 | 可见+可控 | 全开放 |
| Layer 2 流程视图 | 不开放 | 局部 | 完整 | 完整 |
| Preview 预演 | 简化 | 完整 | 高级 | 完整+调试 |
| 质量评分 | 基础摘要 | 标准 | 高级 | 完整可配置 |
| 候选竞技场 | 少量候选 | 完整 A/B/C | 高级淘汰机制 | 全实验模式 |
| 预算治理 | 后台默认 | 项目级可见 | 项目级可调 | 全量策略池 |

## 五、Stage 1 — 工坊 Workbench 详细设计

### 5.1 核心理念

用户面对的不是节点，而是自己的故事。
AI 不应像流程控制器，而应像能随时插话、但不会抢走写作主导权的编剧搭档。

### 5.2 核心对象

- `Story Bible`
- 角色卡
- 世界观 / 设定
- `Beat`
- `Scene`
- 对白草稿
- 情绪线 / 张力线

### 5.3 页面结构

- 左侧：项目导航（Story Bible / 角色 / 场景 / 章节 / Beat / 情感曲线）
- 中间：主编辑区（Beat 编辑、Scene 编辑、对白编辑、结构化视图）
- 右侧：AI 助手面板（建议、改写、扩写、续写、一致性提醒、知识引擎参考）
- 底部：字数、情绪曲线、张力状态、一致性状态、自动保存状态

### 5.4 AI 助手模式

AI 助手不以“弹窗打断”的方式介入，而以“侧栏候选卡 + 行内快捷操作”方式存在。

典型操作：

- 扩写
- 改写
- 精简
- 视觉化
- 换语气
- AI 续写三方向
- 补充场景描写
- 补充潜台词
- 调整对白节奏

### 5.5 知识引擎嵌入方式

知识引擎在工坊阶段不是独立模块，而是被动建议系统：

- 叙事结构参考
- 角色心理学建议
- 对白技巧建议
- 节奏与张力提醒

### 5.6 Normal 自动推进策略

对 `Normal` 用户，系统可自动推进以下步骤：

- 导入小说 / 剧本
- 自动拆章节
- 自动提取角色
- 自动生成初版 Scene 列表
- 自动给出 Story Bible 草案
- 自动建议“进入画板”的时机

### 5.7 向 Stage 2 的过渡

进入画板的条件不是“用户主动去搭节点”，而是：

- Beat / Scene 已足够稳定
- 角色与场景上下文基本明确
- 用户选择“开始拆镜 / 进入可视化生产”

系统随后自动：

- 选择模板
- 建立初始 Shot Card 组
- 带入角色、场景、风格资产
- 打开对应 Scene 的画板

## 六、Stage 2 — 画板 Board 详细设计

### 6.1 核心理念

画板不是节点画布的美化版。
画板是让创作者围绕镜头卡片、参考素材、候选方案和资产锚点进行空间化创作的工作台。

### 6.2 为什么是 TapFlow 式而不是纯 Figma/Miro

- `TapFlow` 的优势是“对象可执行、结果可原位更新、流程可以沉到底层”
- `Figma/Miro` 的优势是评论、Presence、Frame、共享、协作感
- 因此最终采用：**TapFlow 式创作画板 + Figma/Miro 式协作增强层**

### 6.3 基本操作单元：Shot Card

画板的主对象不是 `Node`，而是 `Shot Card`。

一个 `Shot Card` 至少包含：

- 所属 Scene / Beat
- 镜头目标
- 构图 / 调度简述
- 角色状态
- 情绪目标
- 参考素材
- 生成候选
- 质量评分摘要
- 推荐下一步动作

### 6.4 左侧面板：素材与资产面板，而非节点库

左侧默认展示：

- 当前章节 / Scene / Shot 导航
- 角色卡与状态
- 场景资产
- 风格板
- Prompt 包
- 参考图 / 参考视频
- 历史优秀镜头候选

而不是默认把系统做成“拖节点的工程师工作台”。

### 6.5 画板上的卡片类型

- `Shot Card`
- `Reference Card`
- `Asset Card`
- `Mood Board Card`
- `Color Script Card`
- `Performance Card`
- `Comment Card`
- `Result Card`
- `Review Card`

### 6.6 空间语义：邻近 = 关联

画板中一部分关系不必通过显式连线表达，而可以通过空间语义表达：

- 参考图放在 Shot 附近 = 对该 Shot 的视觉参考
- 风格板放在一组 Shot 上方 = 这组镜头共享风格约束
- 角色状态卡贴近某镜头 = 该镜头受该状态影响

系统仍会在后台记录结构化绑定关系，确保不是“看起来有关，实际上没绑定”。

### 6.7 两层画板模型

#### Layer 1：创作视图（默认）

- 无复杂连线
- 以卡片、关系、候选和素材为主
- 面向绝大多数用户

#### Layer 2：流程视图（Pipeline View）

- 显示底层节点、依赖、执行链、回写链
- 用于 `Canvas+` 的高级用户与调试场景
- 不是默认入口

### 6.8 Shot Portal（全屏编辑）

双击 `Shot Card` 可进入 `Shot Portal`。

`Shot Portal` 提供：

- 导演 7 步工作法
- 镜头候选对比
- 角色与情绪控制
- 构图 / 光线 / 运镜参数
- Prompt 原位调优
- Cinema Lab 高级控制（`Hidden+`）

### 6.9 Pipeline 面板（抽屉式）

Pipeline 不再是默认画布，而是可展开的抽屉式引擎盖。

其中包含：

- 节点链
- Agent 任务链
- 运行状态
- 工件版本
- 重跑入口
- 回写入口
- Token / Cost / Duration（高版本）

### 6.10 一致性守护：全局背景服务

一致性守护默认不作为需要用户频繁操作的显式节点，而是全局背景服务。

它持续监测：

- 角色一致性
- 视觉一致性
- 逻辑一致性
- Scene 连续性
- 风格偏移
- 音画冲突

### 6.11 中段的最终默认动作

画板的默认动作顺序应是：

1. 看当前 Shot / Scene 的结果
2. 选候选
3. 比较优劣
4. 重跑局部
5. 写回确认
6. 推送到预演

而不是：

1. 先搭流程图
2. 再理解节点
3. 再配参数
4. 最后才看结果

## 七、Stage 3 — 预演 Preview 详细设计

### 7.1 核心理念

预演不是导出前的检查页，而是导演审片工作台。

它必须让用户真正以“看片”的方式做判断，而不是继续处理抽象数据结构。

### 7.2 页面结构

- 左侧：媒体池、Scene / Shot 列表、标记点、审片任务列表
- 中间：时间线 + 监视器
- 右侧：审核面板（节奏、连续性、声音、字幕、问题列表、回跳入口）
- 底部：轨道、波形、导出队列、版本切换

### 7.3 核心对象

- `Animatic`
- `Timeline`
- `Clip`
- `Subtitle Track`
- `Voice / Music Track`
- `Review Marker`
- `Export Package`
- `CapCut Draft`

### 7.4 导演审片回路（强闭环）

Stage 3 不只是“预览”，而必须支持完整闭环：

**看片 → 批注 → 回跳修正 → 再看片**

具体要求：

- 在时间线或监视器中直接下批注
- 每个批注能定位到来源 `Shot Card` / `Pipeline Step`
- 回跳后修改结果可快速重新送回预演
- 新旧结果可并排看片比较

### 7.5 预演播放引擎

预演引擎应支持：

- 镜头顺序播放
- 临时转场
- 字幕草案
- 语音 / 音乐预放
- 标记点导航
- 候选片段替换
- 批注锚点回放

### 7.6 版本差异

- `Normal`：简化预演
- `Canvas`：完整预演与回跳修正
- `Hidden`：高级审片、声音叙事、Cinema Lab 联动
- `Ultimate`：调试元数据、版本回放、评分面板、AB 预演

## 八、五大新增质量与控制系统（v7.0 核心补强）

这是本次为“奥斯卡级 / 顶级短剧级”目标新增的必备系统。
没有这 5 个系统，平台容易停留在“能生成”，但难以真正稳定地产出高质量作品。

### 8.1 质量评分系统（Six-Dimension Quality Scoring）

#### 目标

不只是生成内容，而是建立一套可解释、可比较、可追踪的质量评价体系。

#### 六维评分

1. **故事 Story**
   - 目标是否清晰
   - 冲突是否有效
   - 情绪推进是否成立
   - 结构节奏是否稳定

2. **表演 Performance**
   - 人物状态是否准确
   - 潜台词是否成立
   - 对手戏张力是否足够
   - 动作与情绪是否匹配

3. **镜头 Cinematography**
   - 构图是否服务叙事
   - 运镜是否必要且有效
   - 景别变化是否合理
   - 视觉重点是否明确

4. **节奏 Pacing**
   - Shot 长度是否合适
   - 信息节奏是否过密/过慢
   - 爽点 / 情绪峰值是否到位
   - 疲劳感是否出现

5. **声音 Sound**
   - 旁白 / 对白是否抢戏
   - 音乐入点是否合理
   - 静默是否有效
   - 声画关系是否成立

6. **连续性 Continuity**
   - 人物形象连续
   - 空间关系连续
   - 视线方向连续
   - 光线 / 风格连续
   - 时间逻辑连续

#### 输出形式

- 每个 `Shot`、`Scene`、`Sequence` 均可生成六维评分雷达图
- 评分附带解释文本、风险点、改进建议
- 评分结果进入候选竞技场与预演审核面板

### 8.2 候选方案竞技场（Candidate Arena）

#### 目标

把“多个候选结果”从杂乱列表，升级为有规则的方案竞技场。

#### 核心能力

- 每个关键 `Scene / Shot` 支持 A/B/C 方案并列
- 支持两两比较、三方比较、盲评、打分、投票
- 支持“淘汰 / 晋级 / 保留观察”三种状态
- 支持从竞技场直接写回当前主版本

#### 适用对象

- 镜头方案
- 表演方案
- 音乐方案
- 转场方案
- 宣发封面 / 海报 / 预告片方案

### 8.3 导演审片回路（Director Review Loop）

#### 目标

把预演台从“看结果”升级为“导演级审片系统”。

#### 核心闭环

- 看：在 `Preview` 中观看当前结果
- 批：对节奏、表演、镜头、声音、连续性做结构化批注
- 跳：回跳到对应 `Shot Card` / `Pipeline Step`
- 修：做局部重跑、候选替换、参数微调
- 再看：返回预演比对新旧版本

#### 必备能力

- 批注与来源对象双向绑定
- 批注分级：致命 / 严重 / 一般 / 建议
- 批注可转化为任务
- 审片历史可回溯

### 8.4 表演与情绪控制层（Performance & Emotion Control Layer）

#### 目标

把“人物状态、潜台词、情绪强度、对手戏张力”变成可控对象，而不只是文本描述。

#### 核心对象

- `Character State`
- `Subtext`
- `Emotion Intensity`
- `Opponent Tension`
- `Inner Objective`
- `Action Beat`

#### 作用位置

- Stage 1：帮助写作和对白打磨
- Stage 2：进入 Shot Card、Prompt、导演控制与表演候选
- Stage 3：进入审片与回跳修正

### 8.5 预算与重跑治理（Budget & Rerun Governance）

#### 目标

高质量不是无限重跑换来的，而是要在预算内做更优决策。

#### 核心规则

- 设定每项目 / 每阶段 / 每类型任务的预算上限
- 设置候选数量上限
- 设置高成本模型使用条件
- 设置自动降级策略与缓存复用策略
- 设置局部重跑优先级，避免整链重跑

#### 可视化能力

- 项目预算仪表盘
- Shot / Scene 成本分布
- 重跑热区识别
- 高浪费路径告警

## 十五、风险与缓解

### 15.1 过度前台简化风险

如果把前台做得太轻，可能损失系统可控性。

缓解：

- 前台简化，后台不简化
- Layer 2 / Pipeline 抽屉始终存在
- Hidden / Ultimate 保留完整系统视图

### 15.2 过度系统化风险

如果让用户总是看到工作流、节点和调试信息，就会破坏心流。

缓解：

- 默认藏复杂度
- 把复杂度变成系统默认值、推荐路径、质量守护和预算规则

### 15.3 高质量高成本风险

追求高质量必然提高模型成本、审核成本和重跑成本。

缓解：

- 预算治理
- 局部重跑优先
- 候选数控制
- 质量评分先筛选、再升级高成本模型

### 15.4 质量系统失真风险

如果六维评分不准，会误导决策。

缓解：

- 人工校准
- 标杆案例对照
- 评分解释可见
- 评分不直接强制替代人类决策

## 十六、渐进式披露策略

`newplan4.md` 作为母规划文档，会比较大。
这本身不是问题，但不能让所有角色都直接从这份文档开始。

建议文档体系如下：

- `newplan4.md`：母规划 / 全量主版本
- 管理层摘要：执行摘要版
- 产品交互稿：三段式 UX 版
- 当前 Phase 指南：阶段执行版
- 模块索引：按系统模块快速跳转

这样既能保留完整性，又不会影响渐进式披露和团队协作效率。

## 十八、最终结论

最终产品路线正式定为：

- 前台采用 `claude-plan.md` 的三段式创作引擎
- 后台采用 `newplan3.md` 的系统级骨架
- 中段默认采用 Shot Card 驱动的 Tapnow 式创作画板
- Pipeline 作为可见但不强迫使用的引擎盖层存在
- 五大新增质量与控制系统贯穿三段式全过程

这条路线的目标不是让用户学会复杂工具，而是让用户在系统的支持下，把绝大部分精力持续放在：

- 故事是否成立
- 人物是否真实
- 镜头是否有力
- 节奏是否有效
- 声音是否加分
- 连续性是否稳定

只有这样，平台才有机会在效率、可控性、质量和创作心流之间取得真正高水平的平衡，并逐步逼近“奥斯卡电影级 / 国内顶级短剧级”的终局目标。
