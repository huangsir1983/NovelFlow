# TapNow 画布网站仿制需求拆解

> 日期: 2026-03-11
> 适用项目: 虚幻造物 / Unreal Make
> 目标: 基于 TapNow 官方文档与公开页面，抽取其“画布型 AI 创作网站”的核心操作，并转化为可研发的软件设计需求
> 说明: 本文聚焦公开可见能力，不覆盖 TapNow 内部计费、风控、运营后台等未公开模块

---

## 1. 研究结论

TapNow 的核心不是单点 AI 生图或生视频，而是一个 **可执行的无限画布工作台**：

1. 用户在画布中以节点方式组织创意、素材、提示词、图片编辑和视频生成。
2. 每个节点既是内容单元，也是可运行单元，运行后原位展示结果。
3. 节点之间通过连线传递上下文，使“文本 → 图片 → 编辑 → 视频 → 海报/素材”形成可复用流程。
4. 用户可以把流程保存为模板、公开分享、被他人克隆 Remix。

如果你要仿制 `https://app.tapnow.ai/`，真正需要做的不是一个“白板”，而是一个 **带运行时、历史记录、素材沉淀、模板市场、分享克隆能力的 AI 画布系统**。

## 2. 产品定位

### 2.1 产品定义

本产品定义为：

**面向图像与视频创作者的 AI-Native 无限画布平台。**

用户可在一个画布中完成：

1. 创意记录与提示词整理
2. 图像生成
3. 图像编辑与精修
4. 视频生成
5. 结果沉淀为素材
6. 工作流模板化
7. 分享、克隆、二次创作

### 2.2 核心价值

1. 把分散的 AI 操作统一进一张可执行画布。
2. 把一次性提示词操作升级为可复用工作流。
3. 把“生成结果”变成“可管理资产”。
4. 把个人创作行为升级为社区 Remix 生态。

---

## 3. 目标用户

### 3.1 主用户

1. AI 图片创作者
2. AI 视频创作者
3. 海报/视觉概念设计师
4. 社交媒体内容创作者
5. 短片导演与分镜创作者

### 3.2 使用场景

1. 从文本提示词开始，逐步生成角色图、场景图、镜头图。
2. 用参考图、多图融合、局部编辑做视觉迭代。
3. 由单图或首尾帧继续生成视频镜头。
4. 把常用工作流保存成模板，团队复用。
5. 浏览社区公开画布，克隆后替换素材继续创作。

## 6. 用户操作到软件需求的映射

本节是核心。每个操作都转成软件设计需求。

### 6.1 账号与进入系统

#### 操作 1: 登录 / 注册

前端需求：

1. 提供邮箱登录、验证码登录或 OAuth 登录。
2. 首次登录后自动进入项目列表。
3. 未登录访问分享页可浏览公开内容，但编辑时要求登录。

后端需求：

1. 用户账户表
2. Session/JWT 鉴权
3. OAuth 绑定能力
4. 访客访问公开画布权限校验

数据需求：

1. `user`
2. `user_profile`
3. `auth_identity`

3. 支持显示项目缩略图、更新时间、公开状态。

### 6.3 无限画布交互

#### 操作 4: 拖动画布、缩放画布、框选节点

1. 支持鼠标拖拽平移画布。
2. 支持滚轮缩放。
3. 支持框选多个节点。
4. 支持快捷键居中、缩放到适合视图。

技术需求：

1. 节点可拖拽。
2. 多选时可一起移动。

4. 节点尺寸改变后应自动刷新连线路径。

### 6.4 节点系统

#### 操作 6: 新建节点

前端需求：

1. 支持通过左侧工具栏创建节点。
2. 支持通过右键菜单创建节点。
3. 支持通过 `/` 快捷命令面板创建节点。
4. 新建节点后自动聚焦输入框或参数面板。

后端需求：

1. 节点类型注册表
2. 节点 schema 配置化
3. 节点默认参数模板

数据需求：

1. `node`
2. `node_type`
3. `node_config_schema`

#### 操作 7: 连线节点

前端需求：

1. 节点之间可拖出连接线。
2. 连接时显示可连接目标高亮。
3. 非法连接需即时阻止并给出提示。
4. 连线后下游节点可引用上游输出。

后端需求：

1. DAG 校验
2. 循环依赖阻止
3. 数据类型兼容校验

数据需求：

1. `edge`
2. `port_schema`

#### 操作 8: 复制、删除、重命名节点

前端需求：

1. 节点支持复制、删除、重命名。
2. 复制节点时可选择是否复制历史结果。
3. 删除节点时连线自动清理。

后端需求：

1. 文档 patch 接口
2. 乐观更新
3. 撤销/重做日志

数据需求：

1. `canvas_operation_log`

### 6.5 文本类节点

#### 操作 9: 创建文本节点

适用场景：

1. 写提示词
2. 写创意说明
3. 写镜头描述
4. 写工作流注释

前端需求：

1. 支持标题 + 正文编辑。
2. 支持多行文本。
3. 支持把文本插入到下游节点参数中。
4. 支持富文本为可选增强，不作为 MVP 必需。

后端需求：

1. 文本节点版本保存
2. 文本变更历史

数据需求：

1. `node_text_payload`

#### 操作 10: 生成灵感和说明

这是 TapNow 文档里明确存在的一类文本能力，可理解为“AI 辅助生成提示词或创意描述”。

前端需求：

1. 提供“帮我写描述/帮我扩写提示词”按钮。
2. 生成结果可一键覆盖、追加、另存为新文本节点。

后端需求：

1. 接入 LLM 文本生成服务
2. 支持多轮重写与风格化

数据需求：

1. `ai_text_run`

### 6.6 图片生成节点

#### 操作 11: 文本生图

前端需求：

1. 节点参数包含 prompt、negative prompt、尺寸、风格、模型、生成数量。
2. 运行后在节点内原位显示图片网格结果。
3. 支持将某张结果设为封面、送入下游节点、保存到素材库。

后端需求：

1. 图像生成任务派发
2. 异步任务轮询或流式状态更新
3. 图片对象存储

数据需求：

1. `generation_run`
2. `generation_asset`
3. `node_output`

#### 操作 12: 单图生成

理解为：以一张参考图作为条件图生成或延展。

前端需求：

1. 节点支持上传或引用一张图。
2. 支持设置参考强度。

后端需求：

1. 图像条件生成接口
2. 文件上传服务

理解为：两张图共同参与生成，用于角色融合、风格融合、主体与场景结合。

前端需求：

1. 节点支持两张输入图。
2. 支持设置双图权重。

后端需求：

1. 多图条件生成

1. 支持 2x、4x 或“高清增强”模式。

### 6.9 运行与结果管理

#### 操作 22: 运行单节点

前端需求：

1. 每个可执行节点必须有独立 Run 按钮。
2. 节点状态包括 idle、queued、running、success、error。
3. 运行中显示进度与取消按钮。

后端需求：

1. 任务队列
2. 状态回调或轮询
3. 失败重试机制

#### 操作 23: Run All

这是 TapNow 公开页面明确强调的关键操作，属于仿制重点。

前端需求：

1. 可从顶部触发整个工作流运行。
2. 应按 DAG 拓扑顺序自动执行。
3. 失败节点需中断下游或允许用户配置“跳过失败继续执行”。
4. 用户可查看当前正在运行到哪个节点。

后端需求：

1. 工作流编排器
2. DAG 拓扑排序
3. 节点缓存命中策略
4. 批量任务状态聚合

数据需求：

1. `workflow_run`
2. `workflow_run_node_status`

#### 操作 24: 查看当前生成和历史生成

前端需求：

1. 用户可查看节点当前输出。
2. 用户可展开历史结果列表。
3. 历史结果支持重新设为当前结果。
4. 历史结果支持另存素材库、复制到其他节点。

后端需求：

1. 按节点查询运行历史
2. 结果版本索引

## 8. 节点统一设计规范

为了让系统可扩展，所有节点必须统一遵守同一套设计规范。

### 8.1 节点通用字段

```ts
type CanvasNode = {
  id: string;
  projectId: string;
  type: string;
  title: string;
  position: { x: number; y: number };
  size?: { width: number; height: number };
  inputPorts: NodePort[];
  outputPorts: NodePort[];
  config: Record<string, unknown>;
  runtime: {
    status: "idle" | "queued" | "running" | "success" | "error";
    currentRunId?: string;
    lastRunAt?: string;
    cacheKey?: string;
  };
  ui: {
    collapsed?: boolean;
    selectedResultId?: string;
  };
};
```

### 8.2 端口类型

```ts
type PortDataType =
  | "text"
  | "image"
  | "images"
  | "video"
  | "videos"
  | "mask"
  | "asset"
  | "prompt"
  | "json";
```

### 8.3 节点 UI 统一要求

每个可执行节点都要有：

1. 标题区
2. 输入区
3. 参数区
4. Run 按钮
5. 结果展示区
6. 历史结果入口
7. 保存素材按钮
8. 错误提示区

## 9. 运行时架构需求

### 9.1 前端运行时

前端需要：

1. 管理画布文档状态
2. 管理选中态、缩放态、展开态
3. 管理节点运行态
4. 管理任务轮询或事件订阅
5. 管理历史结果切换

推荐：

1. React 19 + Zustand
2. `@xyflow/react`
3. 分离 `document state` 与 `runtime state`

### 9.2 后端运行时

后端需要：

1. 统一任务提交接口
2. 多模型 provider 抽象
3. 工作流编排器
4. 任务状态存储
5. 结果资产持久化
6. 配额/并发控制

推荐服务拆分：

1. `canvas-service`
2. `workflow-orchestrator`
3. `generation-service`
4. `asset-service`
5. `template-service`
6. `share-service`

### 9.3 缓存策略

系统必须支持：

1. 同配置重复运行时的缓存命中
2. 上游未变化时跳过重复节点
3. 强制重跑

## 10. 数据库模型建议

建议最小表结构：

1. `users`
2. `projects`
3. `canvas_documents`
4. `nodes`
5. `edges`
6. `workflow_runs`
7. `node_runs`
8. `assets`
9. `asset_collections`
10. `workflow_templates`
11. `share_links`
12. `clone_relations`
13. `operation_logs`

其中关键关系如下：

1. 一个 `project` 对应一个当前 `canvas_document`
2. 一个 `canvas_document` 包含多个 `node` 与 `edge`
3. 一个 `node` 对应多个 `node_run`
4. 一个 `node_run` 产出零到多个 `asset`
5. 一个 `workflow_template` 来源于某个 `canvas_document` 快照

## 12. 非功能需求

### 12.1 性能

1. 200 个节点以内画布仍可流畅缩放与拖拽。
2. 首屏打开 3 秒内完成基础 UI 渲染。
3. 图片缩略图优先展示，原图延迟加载。

### 12.2 稳定性

1. 任务失败需保留错误原因。
2. 页面刷新后运行记录不能丢失。
3. 自动保存至少每 5 秒或每次结构变更后触发。

### 12.3 安全

1. 私密画布默认不可公开访问。
2. 上传资源需做类型与大小校验。
3. 分享链接需支持失效或关闭。

### 12.4 可扩展性

1. 新节点类型应可配置接入，而不是改一套前后端硬编码。
2. 模型 provider 应可替换。
3. 图片、视频、音频能力应可继续扩展。

## 13. 按 Unreal Make 技术栈的落地方案

结合你当前的技术方向，推荐映射如下：

### 13.1 前端

1. `Next.js 15` 负责站点与 SSR 页面
2. `React 19 + Zustand` 负责编辑器状态与运行时状态
3. `@xyflow/react v12` 负责节点画布

3. `PostgreSQL` 存储画布、节点、运行记录、模板、资产
4. `对象存储` 存放图片/视频资源

### 13.3 共享层

建议在 `@unrealmake/shared` 中抽出：

1. 节点 schema
2. 端口 schema
3. 运行状态类型
4. 资产类型
5. 模板类型

## 15. 你这个项目最该优先实现什么

如果目标是“先做出一个像 TapNow 的画布网站”，建议优先顺序不是平均铺开，而是：

1. 先把 `画布 + 节点运行 + 原位结果展示` 做出来
2. 再做 `图片链路`
3. 再做 `视频链路`
4. 最后做 `模板/社区`

因为 TapNow 真正的产品记忆点是：

1. 画布不是展示型，而是执行型
2. 节点不是表单，而是结果容器
3. 分享不是截图，而是可克隆流程

## 16. 与虚幻造物的结合建议

你不是简单复刻 TapNow，而是可以在其画布范式上接入你自己的影视创作能力。

建议分层：

1. 底层交互层: 仿 TapNow 的可执行无限画布
2. 中层能力层: 图片节点、视频节点、素材节点、模板节点
3. 上层领域层: 小说解析、分镜、导演节点、Cinema Lab、资产锁定

也就是说：

**先用 TapNow 的“交互骨架”，再装入虚幻造物自己的“影视大脑”。**
