## 计划

### 1. 本阶段定位

P0 不追求业务功能完整，而是完成“后续所有 Phase 都依赖的底座”。如果 P0 做得不稳，后面每个阶段都会反复返工。

### 2. 交付范围

- 三段产品主形态正式固化：
  - `Workbench`
  - `Board`
  - `Preview`
- 路由与容器底座：
  - `/projects/{id}/workbench`
  - `/projects/{id}/board`
  - `/projects/{id}/preview`
- 状态层底座：
  - `workbenchStore`
  - `boardStore`
  - `previewStore`
  - `pipelineStore`
- 后端骨架接口：
  - `workflow/pipeline`
  - `artifacts/writeback`
  - `collaboration/share`
  - `preview/export`
- 测试底座：
  - backend `pytest`
  - web `vitest`
  - E2E `playwright`
  - 契约测试门禁占位
- 统一阶段文档、命名规则、验收口径

### 3. 本阶段完成定义

- 新功能默认按“三段工作台”进入，不再继续扩张旧单页心智
- 任何新 API 必须先有测试入口和错误码约束
- 任何新路由必须能进入自动化验证
- 旧 `/projects/[id]` 页面继续兼容，但不再作为新能力主落点

## 修复闭环

### 1. 缺陷进入方式

- 自动化测试失败
- 三段路由无法访问
- API schema 变动未同步
- 新增代码绕开测试底座

### 2. 修复顺序

1. 先修测试基建和命令可执行性
2. 再修路由/状态/API 基础骨架
3. 最后处理样式、文案、轻微结构问题

### 3. 修复动作要求

- 任何缺陷先补失败测试，再修实现
- 修完后必须重跑对应测试层，不允许只人工点击
- 如果缺陷暴露了“阶段边界不清”，同步更新本阶段文档

### 4. P0 结束后遗留项

允许遗留：

- 真数据尚未完全接入
- 契约测试仍是占位阶段

不允许遗留：

- 测试命令不可用
- 三段主路由失效
- 新 API 没有测试
