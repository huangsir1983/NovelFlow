## 计划

### 1. 本阶段定位

P1 是“架构融合期”。目标不是做更多页面，而是把三段路由、共享项目真相、版本门控和后端持久层真正接起来。

### 2. 交付范围

- 旧项目页能力逐步拆入三段：
  - Workbench 承接文字与结构编辑
  - Board 承接中段可执行画布
  - Preview 承接预演与导出
- `projectStore` 与四个新阶段 Store 建立清晰边界
- edition 门控收口到统一入口，避免散点判断继续扩散
- 后端四组骨架 API 从内存态升级为持久化结构
- 建立首版前后端契约快照和错误码表

### 3. 关键交付物

- 三段共享项目上下文模型
- Stage Navigation 进入主流程
- pipeline / writeback / collaboration / preview export 持久化数据模型
- 契约测试样板
- 旧单页迁移清单

## 测试

### 1. 先写什么

- Store 边界测试：
  - 阶段切换时状态不串扰
  - 项目真相更新后各阶段正确感知
- API 持久化测试：
  - pipeline run 能落库
  - writeback version 能审计追踪
  - share/clone 权限可验证
- 契约测试：
  - OpenAPI schema snapshot
  - 关键错误码固定

### 2. 覆盖重点

- 旧页与新三段并存期间不出数据分叉
- 阶段切换不丢项目上下文
- 后端返回结构长期稳定
- 版本门控不出现功能泄露

## 验证

### 1. 场景级验证

- 从项目列表进入 `Workbench / Board / Preview` 任一段，项目数据一致
- 更新项目结构后，跨段读取结果一致
- pipeline 和 writeback 记录能在刷新后保留
- 分享链接和克隆动作能跑通基础链路

### 2. 退出门槛

- 三段共享真相模型明确
- 持久化接口替代内存骨架
- 契约快照生成并纳入回归
- 旧页拆分路线确认，新增能力默认落在新三段

## 修复闭环

### 1. 缺陷重点

- 数据重复存储
- 阶段状态串扰
- 接口 schema 漂移
- edition 权限漏拦截

### 2. 修复要求

- 先复现，再加失败用例
- 所有接口修复必须同时更新契约测试
- 涉及旧页迁移的修复，必须验证旧入口未断

### 3. 阶段升级条件

只有当“真数据接线稳定”后，P2 才能放心做 Normal MVP 业务闭环。
