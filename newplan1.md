## 一、升级总纲

### 1.1 核心升级目标

```

  面板式工作台 → AI-Native Node Canvas
  + Cinema Lab电影实验室
  + Flow Marketplace社区Remix

v5.1 → v6.0 升级 (本次):

  + 剪映工程包导出 (视频+音频+字幕+时间线+标记)     ← 后期交给剪映
  + 宣发物料生成 (封面/海报/预告片草稿)              ← 补齐发行环节
  + Animatic动态预览                                ← 低成本节奏验证

  = 从小说/剧本到"可直接用剪映精剪"的完整制作链路
```

### 1.2 核心设计决策

```
决策1: 资产库 — 跨项目可复用素材管理系统
  不是"单项目美术工具", 而是三层素材仓库:
    官方公用资产 (免费) + 私人分享资产 (收费) + 自用资产 + 项目内资产
  存储: 角色/道具/场景/光线/动作参照等 (提示词/图片/视频)
  自己的资产: 可预览/编辑/重生成
  项目节点引用资产库素材 → 下游视觉一致性锚点

决策2: 后期制作 — 交给剪映
  系统不做: 剪辑时间线 / 调色 / 音效 / 配乐 / 混音
  系统输出: 剪映工程包 (Draft文件夹结构)
  包含: 视频片段 + TTS音频 + AI音乐 + 字幕SRT + 时间线草稿 + 导演标记
  用户在剪映中: 精剪 + 调色 + 音效 + 配乐 + 混音 + 导出成片

决策3: 管线边界重新划定
  虚幻造物负责: 叙事设计 + 视觉资产 + AI生成 + 素材组织
  剪映负责: 精剪 + 调色 + 音频后期 + 最终输出
  两者的交接点 = 剪映工程包
```

## 三、新增核心模块 — 资产库管理系统 (Asset Library)

### 3.1 模块定位

**跨项目的可复用素材管理系统。** 存储角色、道具、场景、光线、动作参照等素材（提示词、图片、视频），供项目节点引用，保证视觉一致性和创作效率。

资产库**不是**单项目的美术设计工具，而是**素材仓库 + 素材市场**：
- 用户积累的优质素材可以跨项目复用
- 官方提供公用素材降低创作门槛
- 私人分享素材形成素材交易生态

### 3.2 资产库三层架构

```
资产库 (Asset Library)
│
├── 1. 公用资产 (Public Assets)
│   │
│   ├── 官方公用 (Official) — 免费
│   │   ├── 官方提供的基础素材包
│   │   ├── 通用角色模板 (古装男/现代女/老人/儿童...)
│   │   ├── 经典场景参考 (书房/街道/宫殿/荒野/太空...)
│   │   ├── 光线预设 (三点布光/烛光/霓虹/逆光...)
│   │   ├── 动作参照 (行走/奔跑/打斗/拥抱/对话...)
│   │   ├── Prompt模板 (已验证的高质量提示词)
│   │   └── 随版本更新持续扩充
│   │
│   └── 私人分享 (Community Shared) — 使用收费
│       ├── 用户将自己的资产发布到公用市场
│       ├── 其他用户付费使用 (创作者定价, 平台分成)
│       ├── 评分 + 使用统计 + 预览
│       ├── 可设置: 仅限预览 / 允许引用 / 允许下载编辑
│       └── 归因链: 标注原创作者
│
├── 2. 自用资产 (Personal Assets) — 跨项目
│   ├── 用户自己创建/收集的素材
│   ├── 不绑定任何项目, 所有项目都可引用
│   ├── 可操作: 预览 / 编辑 / 重生成 / 删除
│   ├── 来源:
│   │   ├── 手动上传 (图片/视频/提示词)
│   │   ├── AI生成后保存到自用库
│   │   ├── 从项目内资产"提升"到自用级别
│   │   └── 从公用资产"克隆"到自用库后编辑
│   └── 分类管理: 标签 + 收藏夹 + 搜索
│
└── 3. 项目内资产 (Project Assets) — 绑定项目
    ├── 该项目专属的素材
    ├── 可操作: 预览 / 编辑 / 重生成
    ├── 来源:
    │   ├── 从自用资产/公用资产引用 (保留引用关系)
    │   ├── 在项目中AI生成后自动存入
    │   ├── 手动上传
    │   └── 从知识库角色/场景自动创建空资产卡
    ├── 引用到项目时可二次编辑 (不影响源资产)
    └── 项目内资产 = 下游视觉生成的锚点
```

### 3.3 资产类型体系

```
资产类型 (Asset Types)
├── 角色资产 (Character)
│   ├── 内容: 面部特征/体型/标志特征 (提示词)
│   │         参考图: 正面/侧面/3/4视角 (图片)
│   │         表情参照/动态参照 (图片/视频)
│   ├── 附属: 服装资产 (可挂多套)
│   ├── 附属: 角色视觉演变 (随剧情变化的外观版本)
│   └── 关联: 知识库角色档案
│
├── 道具资产 (Prop)
│   ├── 内容: 道具描述 (提示词) + 参考图 (图片)
│   ├── 标记: 叙事用途 / 是否为视觉母题道具
│   └── 关联: 出现场景列表
│
├── 场景资产 (Location)
│   ├── 内容: 场景描述 (提示词)
│   │         概念图 (图片) — 可含日/夜/天气变体
│   │         氛围参考 (图片)
│   │         空间布局图 (图片)
│   └── 关联: 知识库场景数据
│
├── 光线资产 (Lighting)
│   ├── 内容: 灯光设置描述 (提示词)
│   │         灯光效果参考 (图片)
│   │         Cinema Lab灯光参数 (JSON)
│   └── 应用: 可直接加载到Cinema Lab灯光面板
│
├── 动作资产 (Motion/Pose)
│   ├── 内容: 动作描述 (提示词)
│   │         姿态参考 (图片)
│   │         动作参照 (视频)
│   │         Cinema Lab姿态参数 (JSON)
│   └── 应用: 可直接加载到Cinema Lab动作面板
│
├── 风格资产 (Style)  [Hidden+]
│   ├── 内容: Mood Board (图片拼贴)
│   │         色彩剧本 (色彩走向定义)
│   │         视觉风格指南 (提示词+参考图)
│   └── 应用: 作为全局视觉锚点, 约束Cinema Lab和视觉Prompt
│
└── 通用素材 (Generic)
    ├── 用户自由上传的任意素材
    ├── 自定义标签分类
    └── 不限格式: 提示词/图片/视频
```

### 3.4 资产数据模型

```typescript
// === 资产核心模型 ===

interface Asset {
  id: UUID;
  owner_id: UUID;                    // 创建者
  scope: AssetScope;                 // 资产归属层级
  project_id?: UUID;                 // 仅 scope=project 时有值
  type: AssetType;                   // character | prop | location | lighting | motion | style | generic

  // 基础信息
  name: string;
  description: string;               // 文字描述 / 提示词
  tags: string[];                    // 标签 (搜索/分类用)
  thumbnail?: string;                // 缩略图

  // 素材内容 (一个资产可包含多种内容)
  contents: AssetContent[];

  // 分享设置 (仅 scope=shared 时生效)
  sharing?: {
    price: number;                   // 0=免费, >0=收费 (分)
    permission: "preview" | "reference" | "download";
    use_count: number;
    rating: number;
    rating_count: number;
  };

  // 关联
  source_asset_id?: UUID;            // 如果是从其他资产克隆/引用的
  knowledge_ref?: {                  // 关联知识库实体
    type: "character" | "location";
    id: UUID;
  };

  // 元数据
  created_at: datetime;
  updated_at: datetime;
}

enum AssetScope {
  OFFICIAL = "official",             // 官方公用 (免费)
  SHARED   = "shared",              // 私人分享 (收费)
  PERSONAL = "personal",            // 自用 (跨项目)
  PROJECT  = "project",             // 项目内
}

enum AssetType {
  CHARACTER = "character",
  PROP      = "prop",
  LOCATION  = "location",
  LIGHTING  = "lighting",
  MOTION    = "motion",
  STYLE     = "style",
  GENERIC   = "generic",
}

interface AssetContent {
  id: UUID;
  format: "prompt" | "image" | "video" | "json";
  label: string;                     // "正面参考" / "夜间变体" / "Cinema Lab参数"
  url?: string;                      // 图片/视频文件URL
  text?: string;                     // 提示词/JSON文本
  metadata?: Record<string, any>;    // 格式特定的元数据
}

// === 角色资产附属 ===

interface CharacterCostume {
  id: UUID;
  character_asset_id: UUID;
  label: string;                     // "日常" / "战斗装" / "礼服"
  description: string;
  reference_image?: string;
  scene_context?: string;            // 在什么场景/情节下穿
}

interface CharacterEvolution {
  id: UUID;
  character_asset_id: UUID;
  trigger: string;                   // "受伤" / "时间跳跃5年" / "黑化"
  changes: string;                   // 外观变化描述
  updated_contents: AssetContent[];  // 变化后的参考素材
  sort_order: number;                // 时间线排序
}

// === 项目内资产引用 ===

interface ProjectAssetRef {
  project_id: UUID;
  asset_id: UUID;                    // 引用的资产
  overrides?: {                      // 项目内的二次编辑 (不影响源资产)
    description?: string;
    additional_contents?: AssetContent[];
  };
  locked: boolean;                   // 锁定 = 下游视觉Prompt强制引用
  role: string;                      // 在项目中的角色: "主角" / "主场景" / "关键道具"
}
```

### 3.6 资产库与其他模块联动

| 联动模块 | 联动方式 | 方向 |

### 4.3 draft_content.json 时间线构建逻辑

```python
class CapCutDraftBuilder:
    """从虚幻造物项目数据构建剪映工程文件"""

    def build_draft(self, project: Project) -> dict:
        draft = self._init_draft_structure()

        # 1. 视频轨道: 按分镜顺序排列AI生成片段
        video_track = self._build_video_track(project.shots)

        # 2. 音频轨道1: TTS配音 (对齐到视频片段)
        tts_track = self._build_tts_track(project.shots)

        # 3. 音频轨道2: 背景音乐 (根据声音叙事标注)
        music_track = self._build_music_track(project.sound_narrative)

        # 4. 字幕轨道: SRT导入
        subtitle_track = self._build_subtitle_track(project.shots)

        # 5. 转场: 导演分镜转场建议 → 剪映转场ID映射
        self._apply_transitions(video_track, project.shots)

        # 6. 片段时长: 根据TTS音频时长 + 导演节奏建议
        self._adjust_durations(video_track, tts_track, project.tension_curve)

        return draft

    # 导演分镜转场 → 剪映转场映射
    TRANSITION_MAP = {
        "cut":              "cut",               # 硬切
        "dissolve":         "dissolve",          # 溶解
        "fade_to_black":    "fade_black",        # 淡入黑
        "fade_to_white":    "fade_white",        # 淡入白
        "wipe_left":        "wipe_left",         # 左划
        "wipe_right":       "wipe_right",        # 右划
        "zoom_in":          "zoom_in",           # 推进
        "zoom_out":         "zoom_out",          # 拉远
        "match_cut":        "cut",               # 匹配剪辑→硬切(需手动调整)
        "j_cut":            None,                # J-cut→标记点(需手动音频错位)
        "l_cut":            None,                # L-cut→标记点
        "smash_cut":        "cut",               # 撞击切
        "iris":             "circle",            # 光圈
        "whip_pan":         "slide",             # 甩镜
    }
```

| **P1** | 版本门控复杂度爆炸 | 功能泄漏/锁死 | Phase 0 统一中间件 |

### 18.2 高风险 — 可能导致项目失败

#### R1: Claude API 单点依赖

**现状**: 9个Agent全部依赖 Anthropic Claude API，降级链 `opus → sonnet → haiku` 仍是同一家。

**风险场景**:
- Anthropic API 宕机 / 大规模限流 → 所有Agent停摆，系统核心链路全断
- 价格大幅调整 → 成本模型崩溃
- 政策变化（内容审核收紧/区域限制）→ 部分功能不可用

**缓解策略**:
```python
# Phase 0: AIEngine 做 provider 抽象层
class AIEngine:
    providers = {
        "anthropic": AnthropicProvider,   # 主力
        "openai": OpenAIProvider,         # 备选
        "deepseek": DeepSeekProvider,     # 备选(中文场景)
    }

    async def call(self, prompt, model="sonnet", provider="anthropic",
                   fallback_providers=["openai", "deepseek"], ...):
        for p in [provider] + fallback_providers:
            try:
                return await self.providers[p].call(prompt, model, ...)
            except (APIUnavailable, RateLimitError):
                continue
        raise AllProvidersUnavailable()
```
- **Phase 0 必做**: `AIEngine` 层做 provider 抽象，Prompt 模板与 provider 解耦
- **Phase 3 补充**: 关键Agent（Analyst/Director/Visual）补充 OpenAI/DeepSeek 的 Prompt 适配
- **监控**: API 可用性仪表盘，自动切换阈值（连续3次失败 → 切provider）

#### R2: SQLite → PostgreSQL 迁移断裂

**现状**: Phase 0-2 用 SQLite，Phase 3 切 PostgreSQL。Asset/Shot/CinemaLab 大量 JSONB 字段。

**风险场景**:
- SQLite 的 JSON 函数与 PostgreSQL JSONB 行为差异大（索引/查询/类型转换）
- Phase 2 积累的用户数据迁移失败 → 数据丢失
- 两套数据库的 SQL 方言差异导致隐性BUG

**缓解策略**:
- **推荐方案: Phase 0 直接用 PostgreSQL** — Docker Compose 已规划，PG 启动成本低
- 若坚持 SQLite 起步:
  - 严格使用 SQLAlchemy ORM，禁止裸 SQL
  - JSONB 字段操作统一封装为 service 方法（不在 query 中直接操作 JSON）
  - Phase 2 结束前写迁移脚本 + 全量数据校验测试
  - Alembic migration 从 Phase 0 开始维护

#### R3: 剪映 Draft 格式脆弱性

**现状**: CapCut Draft JSON 格式是逆向工程得来的，无官方文档，剪映更新可能随时改变结构。

**风险场景**:
- 剪映新版本改变 Draft JSON schema → 导出的工程包无法导入
- 不同平台的剪映（PC/Mobile/Web）Draft格式有差异
- 剪映国际版(CapCut)与国内版(剪映)的Draft结构不同

**缓解策略**:
```typescript
// CapCutDraftBuilder 做版本适配层
class CapCutDraftBuilder {
  private formatVersion: "5.9" | "6.0" | "6.1";  // 跟踪剪映版本

  build(project: Project): DraftJSON {
    return this.adapters[this.formatVersion].build(project);
  }
}
```
- `CapCutDraftBuilder` 内部做版本分支，新版本增加 adapter 而非修改现有逻辑
- **集成测试**: CI 中加端到端测试（生成Draft → 模拟导入验证JSON schema）
- **保底方案**: 始终保留 JSON 原生导出（用户手动修复）+ PDF 分镜导出
- 关注剪映开发者社区/逆向工程社区的格式变更情报
- 同时支持 DaVinci Resolve XML 导出作为备选输出格式

### 18.3 中高风险 — 会拖慢交付或降低质量

#### R4: 版本门控复杂度爆炸 (4版本 × 9Agent × 30+节点)

**现状**: 50+ feature flag，`useEdition()` hook 散布在所有组件中。

**风险场景**:
- 门控逻辑 BUG → 付费功能泄漏给免费用户（营收损失）
- 或反过来锁死付费功能（用户投诉）
- 测试排列组合爆炸（4版本 × 30节点 = 120种状态）

**缓解策略**:
```typescript
// 统一收敛到 middleware 层，而非 50 个组件各自判断
// 前端: Next.js middleware 拦截路由
export function middleware(request: NextRequest) {
  const edition = getEdition(request);
  const route = request.nextUrl.pathname;
  if (!ROUTE_PERMISSIONS[route]?.includes(edition)) {
    return NextResponse.redirect('/upgrade');
  }
}

// 后端: FastAPI dependency injection
async def check_edition(min_edition: Edition):
    def dependency(request: Request):
        if request.user.edition < min_edition:
            raise HTTPException(403, f"Requires {min_edition}+ edition")
    return dependency

@app.post("/nodes/cinema-lab/execute",
          dependencies=[Depends(check_edition(Edition.HIDDEN))])
async def cinema_lab_execute(...): ...
```
- **前端**: 路由级门控（middleware.ts）+ 节点级门控（Canvas过滤可用节点列表）— 只这两处
- **后端**: FastAPI Depends 统一拦截 — 业务代码不出现 edition 判断
- **测试**: 每个版本一套 E2E smoke test（4套），覆盖所有节点可用性

#### R5: Agent 事件风暴 / 死循环

**现状**: AgentBus 有4种 Hook 类型，9个Agent形成复杂的事件订阅网络。

**风险场景**:
```
用户编辑内容 → Content Hook(2s debounce) → ConsistencyAgent
→ 发现冲突 → Inter-Agent Hook → ScreenwriterAgent 修正
→ 内容变更 → 又触发 Content Hook → ConsistencyAgent → ...循环
```
- `max_rounds=3` 只防了 Inter-Agent 循环，没防 Content Hook 回弹

**缓解策略**:
```python
class AgentBus:
    # 全局事件去重 + trace_id 防环
    async def publish(self, event: Event):
        # 1. trace_id 防环: 同一 trace 链内不重复触发同一 Agent
        if self._seen_in_trace(event.trace_id, event.target_agent):
            logger.warn(f"Cycle detected: {event.trace_id} → {event.target_agent}")
            return

        # 2. 频率限制: 每Agent每分钟最大触发次数
        if self._rate_exceeded(event.target_agent, max_per_minute=10):
            logger.warn(f"Rate limit: {event.target_agent}")
            return

        # 3. Agent 修正产生的内容变更，标记为 source="agent:xxx"
        #    Content Hook 忽略 agent 来源的变更（只响应用户操作）
        if event.type.startswith("content.") and event.source.startswith("agent:"):
            return  # Agent 自身修正不触发 Content Hook

        await self._dispatch(event)
```
- Content Hook 只响应 `source="user"` 的变更
- Agent 修正产生的变更走 Inter-Agent Hook（已有 `max_rounds=3` 限制）
- 全局 trace_id 贯穿事件链，检测环路
- 每Agent每分钟最大触发次数硬限（默认10次）

#### R6: Celery + Redis 在 Windows 上的兼容性

**现状**: 开发环境 Windows 10，Celery v4+ 官方不支持 Windows，Redis 无官方 Windows 版。

**风险场景**:
- 开发者在 Windows 上无法启动 Celery worker → 后端开发受阻
- Windows 和 Linux 行为差异 → 本地通过但 CI/生产失败

**缓解策略**:
- **推荐方案**: Docker Compose 统一开发环境（已规划）
  ```yaml
  # docker-compose.dev.yml
  services:
    redis:
      image: redis:7-alpine
    celery-worker:
      build: ./backend
      command: celery -A celery_app worker -l info
      depends_on: [redis]
    api:
      build: ./backend
      command: uvicorn main:app --reload
      depends_on: [redis, celery-worker]
  ```
- **备选方案**: 用 `arq`（纯Python异步任务队列，Windows原生兼容）替代 Celery
- 前端开发不依赖后端 → Mock API 模式（`NEXT_PUBLIC_MOCK_API=true`）

#### R7: 前后端类型定义割裂

**现状**: 前端 TypeScript types，后端 Python Pydantic models，Asset/Shot/CinemaLab 数据结构极其复杂。

**风险场景**:
- 后端修改了字段名/类型 → 前端未同步 → 运行时崩溃
- Asset 有 7 种 type × 4 种 scope × 4 种 content format = 112 种组合

**缓解策略**:
```bash
# Phase 0: FastAPI 自动生成 OpenAPI schema → 前端类型自动生成
# backend/
fastapi → /openapi.json

# 前端自动 codegen
npx openapi-typescript http://localhost:8000/openapi.json \
  -o packages/shared/src/types/api.generated.ts

# CI 检查: schema 变更时自动重新生成，PR 中包含 .generated.ts diff
```
- FastAPI + Pydantic → 自动 OpenAPI schema（零额外工作）
- `openapi-typescript` 或 `openapi-ts` 自动生成前端类型
- CI pipeline: schema hash 变更 → 触发 codegen → 自动提 PR
- 共享的枚举值（AssetScope/AssetType/Edition）用 JSON schema 定义，双端生成

### 18.4 中风险 — 需要关注但可控

#### R8: 资产库三级 scope 权限安全

**现状**: 公用资产（免费+付费）/ 自用资产 / 项目资产的权限边界复杂。

**风险场景**:
- 付费资产的文件URL被抓取 → 绕过付费直接下载
- 项目内资产意外暴露给非项目成员
- `AssetSharing.permission: preview|reference|download` 权限提升

**缓解策略**:
- 资产文件 URL 必须是签名短链（S3 presigned URL，有效期15min），不暴露存储路径
- scope 校验统一在 API middleware 层:
  ```python
  async def check_asset_access(asset_id: UUID, user: User):
      asset = await get_asset(asset_id)
      if asset.scope == "project":
          assert_project_member(asset.project_id, user.id)
      elif asset.scope == "personal":
          assert asset.owner_id == user.id
      elif asset.scope == "shared":
          assert_sharing_permission(asset, user, required="download")
      # official: 公开访问
  ```
- `preview` 权限只返回低分辨率缩略图，不返回原始文件
- 审计日志记录所有资产下载行为

#### R9: @xyflow/react v12 大规模画布性能

**现状**: 200+节点 + 每个节点实时显示运行状态/输出预览 + Yjs 多人协作。

**风险场景**:
- 节点数量超过100时 Canvas 帧率骤降
- WebSocket 推送节点状态 + Yjs awareness 更新 → 双重通信压力

**缓解策略**:
- React Flow 虚拟化（仅渲染视口内节点）— 已内置
- 视口外节点折叠为简化占位符（不渲染内容预览）
- WebSocket 消息批量合并（每100ms flush一次，而非逐条推送）
- 节点输出预览懒加载（用户hover或点击时才加载）
- Phase 2 进行压力测试: 200节点 × 4人协作 × 持续Agent执行

#### R10: 外部 AI 服务（Kling/即梦/Suno/Udio）不可控

**现状**: 视频生成、音乐生成依赖多个第三方 API，稳定性/定价/可用性完全不可控。

**风险场景**:
- 某服务下线 → 对应 Generation 节点失效
- API 变更 → 生成质量波动
- 价格上调 → 用户成本激增

**缓解策略**:
```python
# Generation 节点做可插拔适配器模式
class VideoGeneratorRegistry:
    adapters = {
        "kling": KlingAdapter,
        "jimeng": JimengAdapter,        # 即梦
        "runway": RunwayAdapter,        # 备选
        "pika": PikaAdapter,            # 备选
    }

    def get_available(self) -> list[str]:
        return [k for k, v in self.adapters.items() if v.health_check()]
```
- 配置文件定义可用 provider 列表，运行时动态选择
- 每个 adapter 实现统一接口（`generate(prompt, params) → result`）
- 健康检查 + 自动降级（首选不可用 → 切备选）
- 用户设置页可选择偏好的生成服务

#### R11: Monorepo 中 shared 包过重 (95%)

**现状**: 95% 代码在 `@unrealmake/shared`，web 和 desktop 只是薄壳。

**风险场景**:
- shared 任何变更 → web + desktop 都重新构建 → CI 时间膨胀
- 不相关的模块变更导致意外破坏

**缓解策略**:
- shared 内部按功能域拆分 barrel export:
  ```typescript
  // 按域导入，而非全量导入
  import { AssetCard } from '@unrealmake/shared/assets';
  import { CanvasNode } from '@unrealmake/shared/canvas';
  import { CinemaLabPanel } from '@unrealmake/shared/cinema-lab';
  ```
- Turborepo `inputs` 配置过滤无关文件变更:
  ```json
  { "tasks": { "build": { "inputs": ["src/**/*.ts", "src/**/*.tsx"] } } }
  ```
- 长期: 若 desktop 差异大，考虑拆 `@unrealmake/canvas` / `@unrealmake/assets` 独立包

## 三段式交互形态升级补充 2026-03-06 
 
- 在原有 节点画布 Agent Pipeline Cinema Lab 规划基础上 确认主交互采用三段式产品形态。 
- 阶段一 Sudowrite 式工作台 负责故事理解 Story Bible Beat Scene 和对白层创作。 
- 阶段二 Figma 或 Miro 式工作台 负责工作流编排 Shot 规划 资产锁定 Prompt 和局部重跑。 
- 阶段三 Premiere 或 CapCut 式工作台 负责 Animatic Timeline 审片 字幕 音轨 Draft 导出和宣发素材。 
- `Canvas` 以上版本不应直接暴露纯空白自由画布 而应以模板工作流 结果摘要卡和局部重跑作为默认体验。 
- 三段式目标是让用户精力优先集中在内容创作 镜头判断和节奏审片上 而不是流程拼装。 
- 详细设计见 `docs/TRI_STAGE_CREATIVE_UX.md` 与 `docs/CANVAS_COMPLEXITY_AND_FLOW.md`。
