  三、模型特征分析
══════════════════════════════════════════════════════════════════════════

  ── ChatGPT (gpt-5.4) ──────────────────────────────────────────────

  综合评价: ⭐⭐⭐⭐ 稳定可靠的基准模型

  优势:
  - 最一致的输出质量, 各阶段均无异常
  - VFF 使用专业电影术语 (J-Cut, L-Cut, 硬切, 溶解)
  - 角色档案详尽, 含外貌/服装/心理/关系网络
  - 场景描述含标准场景头 (INT./EXT. 格式)

  不足:
  - 场景粒度偏粗 (3章仅5个场景)
  - Token消耗最高 (162K), 成本不占优

  VFF样本特点:
  - 平均长度 ~4,100字/场景
  - 包含完整的画面承接/人物位置/逐镜头分解/速度感/标签
  - 镜头类型丰富: 全景、近景、特写、慢动作

  ── Gemini (3.1-pro) ───────────────────────────────────────────────

  综合评价: ⭐⭐⭐ 高效但深度不足

  优势:
  - Token消耗最低 (105K), 性价比最优
  - 唯一标注 FLASHBACK 的模型
  - 场景分类清晰, 时空标注完整

  不足:
  - 仅识别4个角色 (最少), 遗漏关键配角
  - VFF 篇幅最短, 镜头数最少 (43个)
  - 角色档案深度不够

  VFF样本特点:
  - 平均长度 ~2,350字/场景
  - 简洁有力, 偏现代影像风格
  - 含闪回记忆视效描述 (过曝暗红色前世死亡画面)

  ── Claude (opus-4-6) ──────────────────────────────────────────────

  综合评价: ⭐⭐⭐⭐⭐ 最全面最深入

  优势:
  - 场景覆盖最完整 (19个场景, 3.8x ChatGPT)
  - 输出Token最多 (110K), 但输入Token最少 (3.2K) — 极端效率
  - 角色心理描写最细腻, 含前世/今生双重人格分析
  - 故事弧线完整: 冲突→婚姻→怀孕→分离→对峙 全覆盖
  - 镜头数最多 (181个), 分镜最精细

  不足:
  - API调用次数最多 (54次), 因场景多导致6/7阶段各需18-19次调用
  - 场景拆分可能过细, 部分场景可合并
  - 累计耗时偏长 (1,582s)

  VFF样本特点:
  - 平均长度 ~2,876字/场景
  - 注重角色心理与权力关系的空间化表达
  - 使用主观镜头(POV)、低角度等心理化影像语言

  ── Grok (4.20-beta) ───────────────────────────────────────────────

  综合评价: ⭐ 集成失败, 需修复

  失败原因:
  - Stage 1 分章返回空响应, JSONDecodeError
  - SSE流解析未正确处理 Grok 端点的响应格式
  - Token计数全部为0, 表明响应体未被正确读取

  部分输出(失败前):
  - 生成了3个场景、37个镜头、3个VFF
  - VFF文本极度详尽 (scene_000 达10,904字), 远超其他模型
  - 包含极端暴力场景描述 (自残/流血等)

  修复建议:
  - 检查 model_adapters.py 中 Grok SSE 流处理逻辑
  - 增加空响应防护和重试机制
  - 考虑 fallback 到 openai_compat 协议

══════════════════════════════════════════════════════════════════════════
  四、关键发现
══════════════════════════════════════════════════════════════════════════

  1. 场景粒度差异巨大
     同一小说、同一提示词, Claude 生成19个场景而 ChatGPT/Gemini 仅5个。
     这说明 P04_SCENE_EXTRACT 提示词对"场景边界"的定义不够明确,
     不同模型的理解差距导致下游 Shot/VFF 数量成倍差异。

     → 建议: 在提示词中增加"每N字左右切一个场景"的量化指引,
       或加入参考场景数范围 (如 "3章约产出8-12个场景")。

  2. Token效率差异显著
     Claude: 3,241 输入 → 110,961 输出 (34:1 输出/输入比)
     ChatGPT: 102,814 输入 → 59,880 输出 (0.58:1 输出/输入比)
     Gemini: 73,040 输入 → 32,000 输出 (0.44:1 输出/输入比)

     Claude 通过 Responses API 代理时, 输入Token记录可能不完整
     (system prompt 未计入), 实际输入应更高。但输出产出量确实最大。

  3. VFF V2 纯文本格式验证通过
     三个成功模型均正确输出结构化纯文本 VFF, 包含:
     - 画面承接 (scene transition)
     - 人物位置 (character placement)
     - 逐镜头分解 (shot-by-shot)
     - 速度感 (pacing)
     - 标签 (tags)
     相比R7的JSON格式, V2纯文本更利于人工审阅和后续编辑。

  4. 角色识别一致性
     所有模型均识别出核心角色: 我/高姑娘(女主)、沈词(男主)、父亲
     分歧点: 陆姝仪的角色定位 (ChatGPT=supporting, Gemini=supporting, Claude=antagonist)
     Gemini 遗漏了沈母和陆姝仪两个关键配角。

  5. Grok 集成问题定位
     不是模型能力问题 (Grok 的部分VFF输出质量极高),
     而是 API 适配层的 SSE 解析问题。修复后有望成为有竞争力的选项。

  六、已发现并修复的 Bug (2026-03-09)
══════════════════════════════════════════════════════════════════════════

  Bug 1: 场景排序竞态条件 (影响全部模型)
  ────────────────────────────────────────
  现象: 所有模型的 scene_000 均不是来自 ch_000 (楔子/序幕), 而是来自
        后续章节 (ChatGPT/Gemini/Claude → ch_001, Grok → ch_003)

  具体表现:
    ChatGPT: scene_003(ch_000 提亲观察) scene_004(ch_000 退木雁)
             → 应为 scene_000, scene_001
    Gemini:  scene_000(ch_001) → 实际应排在 ch_000 场景之后
    Claude:  scene_000(ch_001) → 19个场景的顺序全部按完成时间而非章节序
    Grok:    scene_000(ch_003) → 完全错乱

  根因: _stage4_scene_and_beat() 使用 asyncio.gather 并行提取场景,
        共享 scene_order_counter 按任务完成顺序递增, 而非章节文本顺序。
        哪个章节的 API 先返回, 就先拿到低序号的 scene_id。
        ch_000 (834字) > ch_001 (424字) > ch_002 (400字), 短章先返回。

  修复: run_test_round8.py: _stage4_scene_and_beat()
    - 移除 scene_order_counter / scene_lock 共享状态
    - 每个章节记录 _ch_idx (章节索引) 和 _local_idx (章内序号)
    - gather 完成后按 (_ch_idx, _local_idx) 排序, 再统一编号 scene_id
    - 保证 ch_000 的场景永远排在最前

  Bug 2: Grok Stage 1 JSON 解析崩溃 (重试机制失效)
  ────────────────────────────────────────
  现象: Grok pipeline 在 Stage 1 (分章) 崩溃, 报 JSONDecodeError

  根因 (多层):
    a) Grok (grok-4.20-beta) 返回的文本包含嵌套 <think> 标签
       在 JSON 值内部插入了 <think>...</think> 块, 例如:
       "end_marker": "谁说我<think>...调整标记...</think>\n不想嫁？"
       stripping 后残留换行控制字符
    b) 旧代码 json.loads(text) 未使用 strict=False,
       遇到换行控制字符 → JSONDecodeError
    c) _stage1_chapter_split() 的重试循环中, extract_json(resp)
       没有 try/except → 异常直接抛到上层, 重试机制形同虚设

  修复:
    - run_test_round8.py: extract_json()
      * 新增 [Agent N][AgentThink] 标签清除
      * 新增控制字符清理 fallback (移除 \x00-\x1f)
    - run_test_round8.py: _stage1_chapter_split()
      * 给 extract_json 调用加 try/except
      * 解析失败时保存错误信息到 _debug/ 并继续重试
      * 非列表结果的防护处理
    - model_adapters.py: OpenAIChatAdapter
      * Token 字段兼容 prompt_tokens / input_tokens 两种命名
      * 代理不返回 usage 时按字符数粗估 token

  九、结论
══════════════════════════════════════════════════════════════════════════

  本轮测试首次实现四大模型的 Pipeline 对比, 验证了以下核心能力:

  1. VFF V2 纯文本格式可行 — 三个模型均正确生成结构化视频脚本
  2. 多模型抽象层可用 — model_adapters 统一接口成功驱动三个不同后端
  3. Claude 综合表现最优 — 场景覆盖最全(19场景), 输出最丰富(181镜头)
  4. ChatGPT 稳定性最佳 — 各阶段无异常, 输出质量一致
  5. Gemini 性价比最优 — 最少Token完成完整流程
  6. Grok 需要修复 — 能力潜力大但适配层有SSE解析缺陷

  推荐生产配置:
  - 主力模型: Claude (opus-4-6) — 深度覆盖 + 高效输出
  - 备选模型: ChatGPT (gpt-5.4) — 稳定兜底
  - 轻量场景: Gemini (3.1-pro) — 成本敏感时使用
