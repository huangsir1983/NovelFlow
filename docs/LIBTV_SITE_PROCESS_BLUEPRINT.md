## 1. 结论先行

如果你要做一个类似 `LibTV` 的网站，最关键的不是先做一个“视频社区首页”，而是先定义清楚下面这个产品闭环：

1. 用户先在公开内容层看到“结果”
2. 再进入详情层看到“过程”
3. 再把过程克隆到自己的工作空间
4. 在工作空间里继续编辑节点、资产、提示词和生成链路
5. 产出新结果后重新发布，形成二创和复刻网络

也就是说，`LibTV` 这类产品本质上不是“作品站”，而是：

- 结果展示层
- 工作流学习层
- 模板复用层
- 创作执行层
- 社区扩散层

这五层叠在一起，才会形成真正可持续的增长飞轮。

## 2. 信息可信度说明

### 2.1 可核验结论

以下结论来自仓库内可直接读取的资料：

- `TapNow` 是创作引擎与画布工作区
- `TapTV` 是公开社区与共享画布分发层
- 用户可以在画布中创建文本、图片、视频、文档、表格、编辑器等节点
- 节点有输入框、快捷指令、指令优化器、工具栏、执行次数等基本操作
- 用户可以保存个人资产
- 用户可以把个人创作链路保存为个人工作流
- 用户可以分享画布，其他用户可以浏览分享页并克隆后二创
- 存在“生图流程”“一键拉片”“首尾帧生视频”等标准创作路径

### 2.2 推断结论

以下内容是为了帮助你构建“类似 LibTV 网站”而做的产品抽象，不等同于你飞书链接中的逐字原文：

- `LibTV` 的公开网站应当以“结果先行，流程可见，克隆即创作”为核心体验
- `LibTX` 可以理解为站内某种标准化创作流程或创作系统，而不是单一页面
- 社区页、详情页、可视流程页、克隆入口、个人工作台之间应存在强耦合闭环
- 真正拉动增长的关键不是单次生成，而是“复刻率、二创率、模板传播率”

## 3. 产品本质拆解

结合 `TapNow/TapTV` 的可读资料，可以把这类网站理解成两层产品：

### 3.1 引擎层

负责真正创作和生产：

- 画布
- 节点
- 模型调用
- 资产沉淀
- 工作流编排
- 结果生成
- 局部编辑

### 3.2 社区层

负责传播、学习、复刻和扩散：

- 作品展示
- 工作流展示
- 公开详情页
- 克隆与二创
- 创作者主页
- 流量分发
- 排行榜与专题

如果你只做社区层，网站会像一个“结果图站”。
如果你只做引擎层，网站会像一个“纯工作台工具”。
真正接近 `LibTV` 的形态，必须是：

`公开内容分发 + 可见创作流程 + 一键复刻进入工作台`

## 4. 整体流程拆解

下面是更接近你要的网站级全流程。

### 4.1 用户主流程

1. 用户进入首页或探索页
2. 被结果内容吸引，例如封面图、短视频片段、风格化案例、剧情片段
3. 点击进入详情页
4. 在详情页同时看到：
   - 最终作品结果
   - 创作流程摘要
   - 使用了哪些节点/模型/资产
   - 作者说明
   - 克隆按钮
5. 用户点击克隆
6. 系统复制该工作流到“我的工作台”
7. 用户基于自己的素材和提示词进行改造
8. 用户生成新的图片/视频/镜头结果
9. 用户将新作品重新发布到社区
10. 其他用户继续复刻，形成传播链

### 4.2 创作者主流程

1. 创建新画布
2. 添加文本节点，梳理创意/脚本/提示词
3. 添加图片节点或上传参考图
4. 添加视频节点，连接上游文本或图片
5. 使用提示词优化器把自然语言变成更可执行的提示词
6. 配置模型、尺寸、执行次数
7. 执行生成
8. 使用工具栏做局部增强：
   - 抠图
   - 扩图
   - 高清
   - 高级融图
   - 图片编辑器
9. 保存满意结果为个人资产
10. 将整条链路整理为个人工作流
11. 发布工作流或作品到社区

### 4.3 视频向流程

基于现有资料，可抽出两条非常关键的视频流程：

#### 流程 A：参考视频拆解

1. 上传参考视频
2. 对视频节点执行“一键拉片”
3. AI 自动拆出分镜、运镜、景别、镜头内容
4. 用户把拆解结果作为反向学习材料
5. 用户将拆出来的语言用于自己的视频 prompt

#### 流程 B：首尾帧生成视频

1. 准备两张高质量图片
2. 创建视频节点
3. 将首帧图、尾帧图连接到视频节点
4. 选择视频模型
5. 调整高级设置和执行次数
6. 执行生成
7. 获得可用的视频镜头片段

### 4.4 社区传播流程

1. 创作者发布分享页
2. 普通用户浏览分享页
3. 用户查看结果、结构、参数、说明
4. 用户点击克隆
5. 系统复制到“我的画布”
6. 用户换素材或换 prompt 做二创
7. 二创作品再次发布

这个流程的关键不是“浏览”，而是“浏览后立即复刻”。

## 5. 各个部件的使用逻辑

这一部分是最适合直接转成页面和组件设计的。

### 5.1 首页 / 探索页

职责：

- 展示最有吸引力的结果内容
- 帮用户理解平台能做什么
- 把用户导向详情页和克隆

推荐组件：

- Hero 区
- 热门案例卡片流
- 分类入口
- 推荐工作流专区
- 热门创作者
- 今日可复刻专题
- 搜索框

使用逻辑：

- 用户第一眼先看结果，不先看技术说明
- 每张卡片必须有清晰 CTA，例如“查看流程”“一键复刻”
- 卡片信息不宜过深，重点展示结果质量和用途

### 5.2 作品/工作流卡片

职责：

- 承担分发入口
- 在极小面积里同时传递“作品价值”和“复刻价值”

建议字段：

- 封面图或短视频预览
- 标题
- 场景标签
- 创作者
- 使用模型
- 复刻次数
- 点赞/收藏
- “查看详情”
- “克隆”

使用逻辑：

- 对结果型用户，封面吸引点击
- 对工具型用户，模型和流程标签帮助筛选
- 对社区型用户，复刻数是强信号

### 5.3 详情页

职责：

- 把“结果”和“过程”绑定起来
- 让用户在不进入编辑器之前就理解这条链路是否值得克隆

建议分区：

- 结果展示区
- 作者信息区
- 流程概览区
- 节点链路预览区
- 资产/参考说明区
- 模型与参数区
- 评论与问答区
- 克隆 CTA

使用逻辑：

- 顶部优先展示结果内容
- 中段展示流程结构，而不是直接暴露复杂编辑器
- 底部承接评论、相似案例、更多复刻

### 5.4 流程预览区

职责：

- 让用户看懂“这不是黑箱”
- 降低克隆门槛

推荐表现：

- 简化版节点图
- 步骤时间线
- 输入到输出的流程摘要
- 节点类型图例

使用逻辑：

- 默认展示摘要态，不要直接把完整可编辑画布搬上来
- 节点点击后应能展开查看该步骤的说明和产物
- 必须让用户知道：哪一步是文本、哪一步是图像、哪一步是视频

### 5.5 节点系统

职责：

- 是网站背后真正的生产单元
- 把不同模态能力以统一形式组织起来

根据现有资料，节点至少应具备：

- 输入框
- 快捷指令
- 指令优化器
- 工具栏
- 执行次数
- 节点连线
- 结果预览
- 下载或继续编辑

使用逻辑：

- 文本节点负责灵感、脚本、说明和 prompt 初稿
- 图片节点负责静帧、参考图、风格图、场景图
- 视频节点负责镜头生成或参考视频解析
- 连线不是装饰，而是上游结果给下游提供输入
- 节点的价值不只是“生成”，还包括“作为后续节点的依赖”

### 5.6 指令优化器

职责：

- 解决普通用户不会写高质量 prompt 的问题
- 把自然语言翻译成模型更稳定的输入

使用逻辑：

- 应该出现在节点输入附近，而不是放在全局深层页面
- 优化后要允许用户继续编辑，而不是强制替换
- 不同模型应有不同优化器策略

### 5.7 素材工具栏

职责：

- 对生成结果做轻量后处理
- 让用户不必频繁跳出到外部软件

根据资料，核心操作包括：

- 抠图
- 扩图
- 高清
- 高级融图
- 图片编辑器

使用逻辑：

- 工具栏应与节点结果强绑定
- 这些操作最好是“上下文工具”，即选中节点才出现
- 后处理结果应自动回流到当前链路，不要成为游离文件

### 5.8 资产库

职责：

- 把一次性生成结果变成可复用资产
- 提高复用率和一致性

建议资产类型：

- 角色
- 场景
- 风格
- 产品
- 文案
- 镜头参考
- 图像素材
- 视频素材

使用逻辑：

- 用户在满意某个结果时，应能一键存为资产
- 资产应该能再次拖回节点作为输入
- 资产既服务当前项目，也服务跨项目复用

### 5.9 个人工作流

职责：

- 把一次创作过程沉淀为模板化方法
- 从“结果复用”升级到“方法复用”

使用逻辑：

- 用户不是只保存单个结果，而是保存整条链
- 工作流应支持命名、标签、说明、适用场景
- 被保存后的工作流是社区传播的核心对象之一

### 5.10 分享 / 克隆 / 二创

职责：

- 形成平台增长闭环

使用逻辑：

- 分享页负责展示
- 克隆动作负责转化
- 二创机制负责留存与传播

如果没有“克隆到我的工作台”这个动作，网站就会变成纯内容站；
如果没有“发布二创”这个动作，网站就很难形成社区网络。

## 6. 最值得借鉴的产品机制

下面这些机制非常值得你在类似 `LibTV` 的网站里保留。

### 6.1 结果先行

先让用户看到成品，再让用户理解过程。

### 6.2 过程可见

流程不能是黑箱，否则用户只会点赞，不会复刻。

### 6.3 克隆优先

用户进入详情页后最重要的动作不是“收藏”，而是“克隆并改”。

### 6.4 节点化生产

把文本、图片、视频、编辑能力统一为节点，可以天然支持复用和回放。

### 6.5 资产沉淀

用户做得越多，越应该积累成个人资产库，而不是每次从零开始。

### 6.6 工作流沉淀

真正形成壁垒的是“方法库”，不是单张图。

### 6.7 站内闭环编辑

用户尽量少跳出站外软件，站内完成更多编辑和增强动作。

## 12. 最小可行版本建议

如果你想尽快做出第一版，不建议一上来就把所有 AI 工作台能力做全。

### Phase 1：先做站点闭环

目标：

- 跑通“浏览 -> 详情 -> 克隆 -> 发布”闭环

最小范围：

- 首页
- 探索页
- 详情页
- 简化工作台
- 克隆功能
- 发布功能
- 创作者主页

### Phase 2：增强创作深度

目标：

- 让用户真的能在站内做出更复杂内容

补充能力：

- 更多节点类型
- 资产库
- 工作流模板
- 图片编辑器
- 视频流程
- 一键拉片

### Phase 3：增强社区和增长

目标：

- 提高复刻率和二创率

补充能力：

- 专题运营
- 排行榜
- 评论问答
- 创作者激励
- 数据洞察

## 13. 最重要的指标

如果你真要做得像 `LibTV`，建议不要只盯 DAU。

更关键的指标是：

- 详情页到克隆页转化率
- 克隆后成功生成率
- 二创发布率
- 工作流复用率
- 资产复用率
- 工作流完成率
- 每个创作者带来的二创链长度

这些指标比“浏览量”更能说明产品是否形成方法传播。

---

## 14. 对设计稿的直接建议

### 14.1 视觉层面

- 首页一定要强结果视觉，不要像后台系统
- 卡片封面优先视频化或动态化
- 详情页要有“作品感”和“方法感”双重层次

### 14.2 交互层面

- 任何关键内容都应尽量 2 次点击内到达克隆
- 不要在公开详情页直接暴露过重的编辑复杂度
- 工作台要把复杂性藏在默认值和模板中

### 14.3 产品层面

- 把“发布作品”与“发布工作流”区分开
- 把“收藏”与“克隆”区分开
- 把“素材”与“资产”区分开

- 不复制什么

要定义：

- 发布的是作品还是工作流
- 是否允许只公开结果不公开过程
- 是否允许只公开摘要流程
